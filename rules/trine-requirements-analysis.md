# Requirements Analysis Rules (Phase 1.5)

## 핵심 원칙

기획서의 모호한 요구사항을 **구현 전에** 인터랙티브 Q&A로 해소한다.

## Q&A 프로세스

1. 기획서 초기 분석 → 불명확점 식별
2. 질문 과다 판정 (0/1~3/4~5/6+)
3. Q&A 실행 (번호 매기고 한 번에 제시)
4. 매트릭스 생성

## 질문 기준

**질문 대상**: 결과물에 영향을 미치는 불명확점만
- 수용 기준 없거나 모호한 FR
- 상충하는 요구사항
- 우선순위 미명시 핵심 기능
- 측정 기준 없는 NFR

**질문 비대상**:
- 기술적 구현 방식 (AI 자체 판단)
- 이미 명확한 요구사항 확인
- 기획서 범위 밖 제안

## 질문 수 판정

| 질문 수 | 판정 | 동작 |
|:-------:|------|------|
| 0개 | 명확 | Q&A 스킵 → 바로 매트릭스 |
| 1~3개 | 정상 | Q&A → 매트릭스 |
| 4~5개 | 경고 | Q&A + 기획서 보완 권고 |
| 6개+ | 반려 | [STOP] 기획서 반려 |

## 매트릭스 생성

- 프로젝트별 추출 스크립트 실행 (예: `node scripts/extract-requirements.mjs <doc>`)
- 출력: `.specify/traceability/{spec-name}-matrix.json`
- 파싱 실패 시 → AI 추출 모드 + [STOP] Human 확인

## 비표준 기획서 Fallback

1. 추출 스크립트 파싱 실패
2. AI가 FR/NFR 추출 → 표준 형식 중간 파일 생성
3. [STOP] Human 확인 필수
4. 확인 후 재실행
5. session-state에 `requirementSource: "ai-extracted"` 기록
