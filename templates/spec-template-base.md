## 4. 사용자 플로우

> [작성 가이드] 각 주요 기능에 대해 Happy Path, Error Path, Edge Case를 분리하여 작성한다. 플로우 단계는 번호를 매기고, 각 단계에서 사용자 액션과 시스템 반응을 명시한다.

### 4.1 [플로우명 1]

#### Happy Path (정상 흐름)

> [작성 가이드] 사용자가 목표를 달성하는 가장 일반적인 경로를 단계별로 기술한다.

```
1. 사용자: [액션]
   → 시스템: [반응]
2. 사용자: [액션]
   → 시스템: [반응]
3. 완료 → [최종 결과]
```

#### Error Path (에러 흐름)

> [작성 가이드] 예상되는 에러 상황과 시스템의 에러 처리 방식을 기술한다. 사용자에게 보이는 에러 메시지와 복구 경로를 포함한다.

| 에러 상황 | 발생 시점 | 시스템 반응 | 사용자 복구 경로 |
|----------|----------|-----------|---------------|
| [예: 유효하지 않은 입력] | [단계 N] | [에러 메시지 표시] | [수정 후 재시도] |
| [예: 네트워크 오류] | [단계 N] | [재시도 안내] | [자동 재시도 / 수동 재시도] |
| [예: 권한 부족] | [단계 N] | [접근 거부 표시] | [권한 요청 안내] |

#### Edge Case (경계 조건)

> [작성 가이드] 정상도 에러도 아닌 경계 상황을 기술한다. 동시 접근, 빈 데이터, 극단값 등.

| 경계 상황 | 기대 동작 |
|----------|----------|
| [예: 데이터가 0건일 때] | [빈 상태 UI 표시] |
| [예: 동시에 같은 리소스 수정] | [낙관적 잠금 / 충돌 알림] |
| [예: 최대 허용치 초과 입력] | [입력 제한 + 안내 메시지] |

#### 메시지 파일 변경 목록

> [작성 가이드] 변경 대상 메시지 파일 경로를 모두 나열한다. 지원 언어별 1개씩.

| 파일 경로 | 언어 | 변경 유형 |
|----------|:----:|:--------:|
| [예: messages/en.json] | en | 추가/수정/삭제 |
| [예: messages/ko.json] | ko | 추가/수정/삭제 |

#### i18n 검증 체크리스트

- [ ] 신규 페이지/컴포넌트의 모든 사용자 노출 문자열이 메시지 키로 관리되는가
- [ ] 삭제되는 페이지/컴포넌트의 메시지 키가 함께 삭제 대상에 포함되었는가
- [ ] 에러 메시지(404, 409, 410 등)가 메시지 키로 관리되는가
- [ ] placeholder, label, button text, 토스트/알림 메시지가 빠짐없이 포함되었는가
- [ ] 지원하는 모든 언어 파일에 동일한 키가 추가/삭제되었는가

| 대상 | 트리거 | 동작 | 지속 시간 |
|------|--------|------|----------|
| [예: 페이지 전환] | [라우트 변경] | [예: 페이드 인/아웃] | [예: 200ms] |
| [예: 목록 아이템 추가] | [데이터 변경] | [예: 슬라이드 다운 + 페이드 인] | [예: 300ms] |
| [예: 삭제 확인] | [버튼 클릭] | [예: 모달 스케일 업] | [예: 150ms] |

> [작성 가이드] 모션 감소 설정(prefers-reduced-motion) 대응 여부도 명시한다.

## 10. 테스트 요구사항

> [작성 가이드] 각 테스트 레벨에서 무엇을 검증하는지, 구체적 시나리오 예시를 포함하여 작성한다. 시나리오는 Given-When-Then 형식을 권장한다.

### 10.1 단위 테스트

> [작성 가이드] 비즈니스 로직, 유틸리티 함수, 데이터 변환 등 독립적으로 검증 가능한 단위를 대상으로 한다.

#### 핵심 로직 테스트

| 테스트 대상 | 시나리오 | 기대 결과 |
|-----------|---------|----------|
| [예: 가격 계산 함수] | 정상 입력 (수량 3, 단가 1000) | 3000 반환 |
| [예: 가격 계산 함수] | 할인율 적용 (10%) | 2700 반환 |
| [예: 입력 유효성 검증] | 빈 문자열 입력 | 검증 실패 반환 |

#### Edge Case 테스트

| 테스트 대상 | 경계 조건 | 기대 결과 |
|-----------|----------|----------|
| [예: 가격 계산 함수] | 수량 0 | 0 반환 (에러 아님) |
| [예: 가격 계산 함수] | 음수 수량 | 검증 에러 발생 |
| [예: 목록 정렬] | 빈 배열 | 빈 배열 반환 |
| [예: 날짜 파서] | 잘못된 형식 입력 | null 반환 또는 예외 |

### 10.2 통합 테스트

> [작성 가이드] 모듈 간 연동, API 엔드포인트, 데이터베이스 연동을 검증한다.

#### API 정상 응답 테스트

| 엔드포인트 | 시나리오 | 요청 | 기대 응답 |
|-----------|---------|------|----------|
| [예: POST /items] | 정상 생성 | 유효한 요청 본문 | 201 + 생성된 리소스 |
| [예: GET /items/:id] | 존재하는 리소스 조회 | 유효한 ID | 200 + 리소스 데이터 |
| [예: GET /items] | 필터 조건 조회 | 쿼리 파라미터 | 200 + 필터된 목록 |

#### API 에러 응답 테스트

| 엔드포인트 | 시나리오 | 요청 | 기대 응답 |
|-----------|---------|------|----------|
| [예: POST /items] | 필수 필드 누락 | 불완전한 요청 본문 | 400 + 검증 에러 상세 |
| [예: GET /items/:id] | 존재하지 않는 리소스 | 잘못된 ID | 404 + 에러 메시지 |
| [예: DELETE /items/:id] | 권한 없는 사용자 | 유효한 ID, 무권한 토큰 | 403 + 권한 에러 |

#### 데이터 연동 테스트

| 시나리오 | 검증 대상 | 기대 동작 |
|---------|----------|----------|
| [예: 아이템 생성 후 조회] | 생성 → 조회 일관성 | 생성한 데이터가 조회 결과에 포함 |
| [예: 트랜잭션 롤백] | 부분 실패 시 데이터 일관성 | 이전 상태로 복원 |

### 10.3 E2E 테스트 (해당 시)

> [작성 가이드] 사용자 관점에서 주요 시나리오를 end-to-end로 검증한다. 섹션 4의 사용자 플로우와 대응시킨다.

#### 주요 사용자 시나리오

| 시나리오명 | 사전 조건 | 사용자 액션 흐름 | 검증 포인트 |
|----------|----------|----------------|-----------|
| [예: 아이템 생성 완료] | 로그인된 상태 | 목록 → 생성 폼 → 입력 → 제출 → 결과 확인 | 목록에 새 아이템 표시, 성공 알림 |
| [예: 검색 후 상세 조회] | 데이터 존재 | 검색어 입력 → 결과 클릭 → 상세 확인 | 올바른 상세 데이터 표시 |
| [예: 에러 복구 흐름] | 서버 에러 상황 | 에러 화면 → 재시도 → 정상 결과 | 재시도 후 정상 동작 |

#### 모바일 디바이스 E2E 시나리오 (프론트엔드 포함 시)

> [작성 가이드] 주요 사용자 시나리오를 모바일 뷰포트에서 반복 검증한다. Playwright `devices` 프리셋을 활용한다.
> 모바일 전용 UI(Bottom Nav, Hamburger, 터치 제스처 등)가 있으면 해당 인터랙션도 검증한다.

| 시나리오명 | 디바이스 | 검증 포인트 |
|----------|---------|-----------|
| [예: 모바일 네비게이션] | iPhone 14 (390x844) | Bottom Nav 표시, 탭 전환 동작, 활성 상태 표시 |
| [예: 모바일 폼 입력] | iPhone 14 (390x844) | 키보드 올라올 때 입력 필드 가림 없음, 제출 동작 정상 |
| [예: 반응형 레이아웃] | Pixel 7 (412x915) | 단일 컬럼 레이아웃, 카드 풀와이드, 데이터 테이블 → 카드 변환 |
| [예: 가로 모드] | iPhone 14 Landscape | 레이아웃 깨짐 없음, 네비게이션 접근 가능 |
| [예: 터치 인터랙션] | iPhone 14 (390x844) | 스와이프 동작, 터치 타겟 탭 가능, 제스처 대체 버튼 동작 |

**Playwright 모바일 디바이스 설정 예시**:
```
// playwright.config.ts
import { devices } from '@playwright/test';

projects: [
  { name: 'Desktop', use: { ...devices['Desktop Chrome'] } },
  { name: 'Mobile iPhone', use: { ...devices['iPhone 14'] } },
  { name: 'Mobile Android', use: { ...devices['Pixel 7'] } },
  { name: 'Mobile Landscape', use: { ...devices['iPhone 14 landscape'] } },
]
```

### 10.4 프론트엔드 테스트 (해당 시)

> [작성 가이드] 프론트엔드 고유의 테스트 패턴을 정의한다. 컴포넌트 렌더링, 사용자 인터랙션, 커스텀 훅 등.

#### 컴포넌트 테스트

| 컴포넌트 | 테스트 시나리오 | 검증 포인트 |
|---------|--------------|-----------|
| [예: 아이템 카드] | Props에 따른 렌더링 | 필수 정보 표시, 조건부 요소 토글 |
| [예: 폼 컴포넌트] | 유효성 검증 인터랙션 | 에러 메시지 표시, 제출 버튼 비활성화 |
| [예: 빈 상태 컴포넌트] | 데이터 없을 때 렌더링 | CTA 버튼 표시, 안내 메시지 |

#### 반응형/모바일 컴포넌트 테스트

> [작성 가이드] 반응형 동작이 있는 컴포넌트에 대해 뷰포트별 렌더링을 검증한다.
> `window.matchMedia` 모킹 또는 컨테이너 크기 조절로 breakpoint 전환을 테스트한다.

| 컴포넌트 | 뷰포트 | 검증 포인트 |
|---------|--------|-----------|
| [예: 네비게이션] | Mobile (< 640px) | Hamburger/Bottom Nav 표시, 데스크톱 메뉴 숨김 |
| [예: 데이터 테이블] | Mobile (< 640px) | 카드 리스트로 변환, 스크롤 없이 데이터 접근 가능 |
| [예: 사이드바] | Mobile (< 640px) | 숨김 상태, 토글 버튼 표시 |

#### 훅/로직 테스트

| 훅/로직 | 테스트 시나리오 | 검증 포인트 |
|--------|--------------|-----------|
| [예: 데이터 조회 훅] | 로딩 → 성공 상태 전환 | 상태값 변화, 데이터 반환 |
| [예: 데이터 조회 훅] | 로딩 → 에러 상태 전환 | 에러 객체 반환, 재시도 함수 제공 |
| [예: 디바운스 입력 훅] | 빠른 연속 입력 | 마지막 입력만 콜백 실행 |

### 10.5 테스트 데이터 전략

> [작성 가이드] 테스트에 사용할 데이터의 생성/관리 방식을 정의한다.

| 항목 | 전략 | 비고 |
|------|------|------|
| 테스트 데이터 생성 | [예: 팩토리 패턴 / 시드 데이터 / 빌더 패턴] | [데이터 간 관계 포함 여부] |
| 데이터 격리 | [예: 테스트별 독립 DB / 트랜잭션 롤백] | [병렬 실행 안전성] |
| 외부 의존성 | [예: API 모킹 / 테스트 서버 / 인메모리 DB] | [모킹 범위와 기준] |
| 민감 데이터 | [예: 가상 데이터 사용, 실 데이터 금지] | [보안 요구사항 연계] |
