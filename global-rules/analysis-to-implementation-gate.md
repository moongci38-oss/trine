# 분석→구현 전환 게이트 (Analysis-to-Implementation Gate)

> 분석/리뷰/진단 작업 완료 후 코드 수정으로 전환하는 시점에 반드시 Human 확인을 거친다.

## 핵심 원칙

**분석 결과를 근거로 곧바로 코드를 수정하지 않는다.**

리포트/리뷰/진단 작성 = 현재 작업 완료. 수정 구현은 별도 개발 사이클로 취급한다.

## 적용 범위

| 분석 작업 유형 | 예시 | 전환 게이트 |
|---------------|------|:---:|
| UI/UX 품질 리포트 | FAIL 항목 수정 | **필수** |
| 코드 품질/복잡도 리포트 | 리팩토링 필요 | **필수** |
| 보안 진단 리포트 | 취약점 수정 | **필수** |
| 코드 리뷰 결과 | 리뷰어 피드백 반영 | **필수** |
| 독립 분석/갭 분석 | 성능 진단, 아키텍처 검토 | **필수** |

## 예외 — 즉시 수정 가능

- 기존 개발 사이클 내 자동 수정 루프 (Lint/Build 실패, CI 실패 등)
- PR 리뷰 코멘트 반영 (동일 PR 범위 내 반복 수정)
- Hotfix 긴급 대응

## 전환 게이트 흐름

```
분석/리뷰/진단 완료 → 리포트 작성 → [STOP]
  → Human에게 결과 보고 + 수정 여부 확인
  → Human 승인 → 프로젝트 개발 워크플로에 따라 수정 사이클 시작
```

## AI 행동 규칙

1. 분석/리뷰/진단 작업이 끝나면 **리포트만 작성**하고 멈춘다
2. FAIL 항목을 발견해도 곧바로 수정 코드를 작성하지 않는다
3. Human에게 결과를 보고하고 수정 여부를 확인받는다
4. Human이 수정을 요청하면 해당 프로젝트의 개발 워크플로(Spec, Plan 등)에 따라 진행한다

## 금지 사항

- 리포트 작성 직후 "수정 Plan"을 바로 작성
- AskUserQuestion으로 "어떻게 수정할까요?" 받고 개발 워크플로 없이 구현
- "간단한 수정이니까 워크플로 없이" 합리화
- 분석 결과를 근거로 Human 확인 없이 코드 변경
